# Серверная часть BarberTime

В проекте нет отдельного backend-приложения. Во время разработки API эмулируется внутри dev-сервера Vite и доступно по префиксу `/api` на том же адресе, что и React-приложение.

Серверная часть состоит из трёх Vite-плагинов, подключённых в `vite.config.ts` в следующем порядке:

1. `authPlugin()` — регистрация, вход, восстановление сессии и выход;
2. `bookingsPlugin()` — создание, просмотр и отмена записей;
3. `vite-plugin-json-server` — публикация данных из `server/db.json`, в текущем приложении используется для получения списка услуг.

Порядок важен: сначала запрос обрабатывают пользовательские middleware. Если маршрут им не принадлежит, вызывается `next()`, после чего запрос передаётся `json-server`.

## Запуск

Из корня проекта выполните:

```bash
npm install
npm run dev
```

После запуска интерфейс и API будут доступны на одном origin, обычно:

```text
http://localhost:5173
http://localhost:5173/api
```

Отдельно запускать сервер не требуется.

> Серверная эмуляция работает только при `npm run dev`. Команда `npm run build` собирает только клиентское приложение, а `npm run preview` не поднимает middleware из `configureServer` и не предоставляет API.

## Структура папки `server`

```text
server/
├── auth.ts             # работа с пользователями, паролями, JWT и cookie
├── authPlugin.ts       # маршруты /api/auth/*
├── bookingsPlugin.ts   # маршруты /api/bookings*
├── db.json             # файловое хранилище данных
└── README.md
```

## Хранение данных

Данные находятся в `server/db.json`:

- `services` — услуги барбершопа;
- `users` — зарегистрированные пользователи;
- `bookings` — созданные записи.

Идентификаторы хранятся в поле `_id`:

- у услуг используются строковые значения (`"1"`, `"2"` и т. д.);
- для пользователей и записей создаются UUID через `crypto.randomUUID()`.

Регистрация, создание записи и отмена записи изменяют `server/db.json` напрямую с помощью `fs.readFileSync` и `fs.writeFileSync`.

Пароли не хранятся в открытом виде: перед сохранением они хешируются библиотекой `bcryptjs` с коэффициентом сложности `10`. В ответах API поле `password` не возвращается.

## Авторизация

Авторизация реализована с помощью JWT в `HttpOnly` cookie.

После успешной регистрации или входа сервер:

1. создаёт JWT с полем `userId`;
2. подписывает его секретом, указанным в `server/auth.ts`;
3. устанавливает cookie `authToken` сроком на 7 дней.

Cookie имеет параметры:

```text
HttpOnly; Path=/; SameSite=Lax; Max-Age=604800
```

Клиентский JavaScript не может прочитать `authToken`. Браузер отправляет cookie автоматически. Общий экземпляр Axios в `src/Api/api.ts` дополнительно настроен с `withCredentials: true`.

В текущей реализации роль хранится в пользователе и возвращается клиенту, однако отдельные права для `user` и `admin` не реализованы. Фактическая ролевая модель приложения — гость и авторизованный пользователь.

## Эндпоинты авторизации

### `POST /api/auth/register`

Создаёт пользователя, сразу авторизует его и устанавливает cookie.

Тело запроса:

```json
{
  "name": "Иван",
  "email": "ivan@example.com",
  "phone": "+79999999999",
  "password": "password123"
}
```

Успешный ответ: `201 Created`.

```json
{
  "user": {
    "_id": "uuid",
    "name": "Иван",
    "email": "ivan@example.com",
    "phone": "+79999999999",
    "role": "user"
  }
}
```

Возможные ошибки:

- `400` — отсутствует обязательное поле или тело запроса некорректно;
- `409` — пользователь с таким email уже существует.

Email перед сохранением очищается от пробелов и приводится к нижнему регистру.

### `POST /api/auth/login`

Проверяет email и пароль, после чего устанавливает новую cookie с JWT.

Тело запроса:

```json
{
  "email": "ivan@example.com",
  "password": "password123"
}
```

Успешный ответ: `200 OK` с объектом `user` без пароля.

Возможные ошибки:

- `400` — email или пароль не переданы;
- `401` — пользователь не найден или пароль неверный.

### `GET /api/auth/session`

Восстанавливает пользовательскую сессию при загрузке приложения.

Ответ для действующей cookie:

```json
{
  "user": {
    "_id": "uuid",
    "name": "Иван",
    "email": "ivan@example.com",
    "phone": "+79999999999",
    "role": "user"
  }
}
```

Если cookie отсутствует, токен недействителен или пользователь больше не существует, маршрут возвращает `200 OK`:

```json
{
  "user": null
}
```

### `POST /api/auth/logout`

Завершает сессию, устанавливая для `authToken` значение с `Max-Age=0`.

Успешный ответ: `200 OK`.

```json
{
  "message": "Выход выполнен"
}
```

## Услуги

### `GET /api/services`

Публичный маршрут. Возвращает массив услуг из коллекции `services` в `server/db.json`.

Этот маршрут обслуживает `vite-plugin-json-server`. Клиент использует его на странице услуг и при подготовке записей для личного кабинета.

## Записи

Маршруты записей реализованы в `bookingsPlugin.ts`.

### `POST /api/bookings`

Создание записи доступно как гостю, так и авторизованному пользователю.

Тело запроса:

```json
{
  "serviceId": "1",
  "date": "2026-07-20",
  "time": "10:00",
  "name": "Иван",
  "phone": "+79999999999"
}
```

Сервер создаёт запись со следующими дополнительными полями:

- `_id` — UUID;
- `status` — `"new"`;
- `userId` — ID текущего пользователя или `null`, если запись создаёт гость.

Успешный ответ: `201 Created` с созданной записью.

Возможные ошибки:

- `400` — не заполнено одно из обязательных полей;
- `401` — cookie присутствует, но содержит недействительный JWT.

### `GET /api/bookings`

Доступен только авторизованному пользователю. Возвращает только те записи, у которых `userId` совпадает с `userId` из JWT.

Возможная ошибка:

- `401` — пользователь не авторизован или токен недействителен.

Гостевые записи имеют `userId: null` и не отображаются в личном кабинете после последующей авторизации.

### `PATCH /api/bookings/:bookingId/cancel`

Отменяет запись текущего пользователя, меняя её статус на `"cancelled"`.

Отменить можно только запись со статусом `"new"` или `"confirmed"`.

Возможные ошибки:

- `400` — некорректный ID или запись уже завершена/отменена;
- `401` — пользователь не авторизован или токен недействителен;
- `404` — запись не найдена либо принадлежит другому пользователю.

## Как клиент работает с cookie

Клиенту не нужно получать, сохранять или добавлять JWT в заголовок `Authorization`. Все запросы выполняются через экземпляр Axios с базовым адресом `/api`:

```ts
const api = axios.create({
  baseURL: "/api",
  withCredentials: true,
});
```

Пример запроса:

```ts
const response = await api.get("/bookings");
```

Cookie будет добавлена браузером автоматически.

## Ограничения текущей реализации

Эта серверная часть предназначена только для учебной разработки и не является полноценным безопасным backend-приложением.

1. API существует только внутри Vite dev-сервера и не попадает в production-сборку.
2. JWT-секрет записан прямо в `server/auth.ts`, а cookie не имеет флага `Secure`.
3. Сервер проверяет в основном наличие полей. Проверка формата телефона, даты, времени и существования услуги выполняется преимущественно на клиенте.
4. Операции чтения и записи файла выполняются синхронно и без защиты от конкурентных изменений.
5. `vite-plugin-json-server` публикует коллекции из `db.json` автоматически. Поэтому кроме описанных маршрутов могут быть доступны необработанные маршруты коллекций, включая `users` и `bookings`.
6. Пользовательские middleware сравнивают `req.url` со строкой целиком. Например, `/api/bookings?x=1` не совпадает с `/api/bookings` и может перейти в обработчик `json-server`, минуя пользовательскую проверку доступа.
7. `json-server` хранит собственное состояние в памяти, а пользовательские плагины записывают изменения напрямую в файл. После изменения `db.json` Vite-плагин перезагружает данные, но между этими действиями возможна кратковременная рассинхронизация.

Поэтому текущую реализацию следует использовать только локально для демонстрации работы React-приложения, JWT-cookie и асинхронных API-запросов.
